# Tally-форма регистрации — структура и контент (одна форма навсегда)

Это заменяет вчерашний файл `gdpr-consent-text-4lang.md` — там было правовое основание "согласие", здесь — правильное "исполнение договора", взятое из твоего реального документа `GDPR_Zpracování_osobních_údajů_2026`. Тот файл можно удалить.

## Принцип: одна форма, бесконечно много событий

Форма создаётся в Tally один раз. Какое именно событие — определяется не созданием новой формы, а скрытыми полями, которые прилетают в URL (его генерирует Apps Script автоматически на каждую строку в Events). Когда меняется состав мероприятий — ничего в самой форме менять не нужно.

## Скрытые поля (Hidden fields в Tally)

⚠️ **Важно для чекбоксов (пункты 6, 7 ниже):** в Tally чекбокс технически — это группа с одной или несколькими опциями, и вебхук присылает значение как **массив ID выбранных опций**, а не `true/false`. Чтобы код в `Code.gs` (функция `isChecked()`) мог однозначно найти нужное поле по названию, делай так: название/вопрос самого чекбокс-блока — короткий идентификатор (`Photo_Consent`, `Waiver_Accepted`), а единственная опция внутри — сам длинный текст согласия. Тогда `byLabel['Photo_Consent']` в коде получит ровно то, что нужно.

- `event_id`
- `title`
- `category`
- `community`
- `price`
- `is_free` — `"1"` если Price_CZK = 0, иначе `"0"` (Code.gs вычисляет это сам)

Все шесть подставляются автоматически из ссылки `Tally_Form_URL`, которую генерирует скрипт. В Tally в настройках Hidden Fields просто заведи поля с этими именами — система сама подхватит значения из `?event_id=...&title=...&category=...&community=...&price=...&is_free=...`.

## Видимые поля и условная логика

⚠️ **Критично для полей ввода ниже (не для текстовых блоков-пояснений вроде вейверов/GDPR/отмены — там 4 языка стопкой правильны).** Вебхук Tally отправляет в `label` ровно тот текст, что стоит заголовком блока — байт в байт, без исключений. Отдельного "внутреннего имени" не существует: функция переименования в Tally меняет только то, что видишь ты в своей панели, и явно не влияет на данные, уходящие в интеграции. Поэтому:
- **Заголовок поля = точная строка ниже** (`Full_Name`, `Telegram_Handle` и т.д.), один в один с тем, что ждёт `byLabel[...]` в `Code.gs`. Респондент увидит именно эту техническую строку — это нормально, не баг вида.
- **Перевод на RU/CS/UK — отдельным текстовым блоком НАД полем**, не частью заголовка. Если стопкой языков заменить заголовок — `byLabel['Full_Name']` не найдёт ничего, поле в Registrations будет молча оставаться пустым, без ошибки, которую сразу заметишь.

⚠️ **Чего лишаемся, уходя от Stripe к личной Revolut-ссылке** (коротко, без уговоров — ты уже принял решение, но это должно быть явным, а не молчаливым):
- **Нет автоматического подтверждения оплаты.** Stripe умеет сам помечать регистрацию в Tally как "оплачено"; Revolut.me такой интеграции с Tally не имеет — сверка "кто заплатил" остаётся на тебе вручную (смотришь приложение Revolut, сверяешь с списком в `Registrations`).
- **У личной ссылки revolut.me есть потолок:** не больше **20 платежей в неделю** и **£250-эквивалент в неделю** через карту (актуально проверь в своём приложении — Revolut меняет лимиты по тарифным планам). При 30 событиях/месяц с активной коммерческой активностью обоих клубов это реалистично может быть достигнуто в пиковую неделю (например, перед стартом сезона, когда многие платят разом) — тогда ссылка перестанет принимать платежи до сброса лимита. Если это начнёт происходить — у Revolut есть отдельный продукт **Revolut Business Payment Links** без такого низкого потолка (до £50,000 за ссылку), но это уже отдельный бизнес-аккаунт, не личный.
- Комиссию с полученных платежей Revolut всё равно берёт (на картах: около 1% + 20p за внутренние платежи, больше — за международные/коммерческие карты) — просто не через Tally/Stripe, а напрямую внутри Revolut.

## Что меняется в Registrations sheet и Code.gs

Новые колонки: `Age`, `Guardian_Name`, `Guardian_Phone`, `Guardian_Relationship`, `Child_Name`, `Child_Age`, `Emergency_Contact_Name`, `Emergency_Contact_Phone`, `Medical_Conditions`, `Equipment_Note`, `Photo_Consent`, `Waiver_Accepted`, `Submission_ID` — добавлены между `Phone_Number` и `Payment_Reference` (плюс `Submission_ID` в конце, для защиты от дублей при повторной доставке вебхука). `Child_Name`…`Equipment_Note` заполняются только для Children's Quest / LARP Battle — для остальных категорий останутся пустыми, это ожидаемо, не ошибка. Обновлённые `Code.gs` и `events-registrations-template.xlsx` — отдельными файлами.

## Как отметить поле обязательным

Там, где в списке выше стоит "обязательное" (Full_Name, Age, Child_Name, Child_Age) — это не появляется само по себе. Выдели блок → Ctrl/Cmd+Shift+R, либо через ⋮⋮ на блоке → toggle "Required". Без этого шага форма отправится и с пустыми этими полями.

## QA-чеклист перед публикацией

Логика в этой форме — самое вероятное место тихого сбоя (данные просто не придут, без ошибки). Пройди форму 5 раз с разными комбинациями, каждый раз проверяя итоговую строку в Registrations:

1. **Age 25, category HEMA Training** → Guardian-блок НЕ появился, вейвер спарринга появился
2. **Age 15, category Hike** → Guardian-блок появился, обычный вейвер (не спарринг)
3. **Age 30, category Children's Quest** → появились Child_Name/Child_Age/Emergency_Contact_*/Medical_Conditions, Guardian-блок не появился (регистрант — взрослый родитель)
4. **category Board Games, price 0** → payment-блок показывает "добровольный взнос", НЕ "вход свободный" и НЕ "оплатите 0 Kč"
5. **category Free Event** → политика отмены скрыта, payment-блок показывает "вход свободный"

Если что-то из этого не сработало — почти всегда причина одна из двух: заголовок поля не совпадает с тем, что ждёт `byLabel` в коде (см. предупреждение выше), или логика "Hide" не была применена к блоку ДО того, как на него повесили условие "Show".

## Текст кнопок (Next/Submit)

Кнопки — не текстовый блок, редактируются отдельно: клик ⚙️ слева от самой кнопки (не через общие Settings формы) открывает поле для правки текста. По умолчанию Tally ставит английский ("Next"/"Submit") без адаптации под остальные языки — то же правило стека языков, что и везде в форме:

- Кнопка перехода на след. страницу (если форма разбита на страницы): `Next / Далее / Další / Далі`
- Финальная кнопка отправки: `Submit / Отправить / Odeslat / Надіслати`
