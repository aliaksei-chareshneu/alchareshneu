## 3. Что система делает автоматически (нулевые трудозатраты)

**На каждое новое событие (одно действие — Status → Trigger_Sync):**
- Создаёт событие в Google Calendar с правильным цветом по клубу/категории
- Публикует анонс в Telegram-канал с inline-кнопкой "Записаться"
- Генерирует правильную Tally-ссылку с предзаполненными параметрами
- Пишет в личку администратору готовые тексты для WhatsApp и Facebook
- Обновляет карточку на сайте (появляется автоматически через JSONP + кэш)
- Сохраняет ссылку на созданное событие и ID поста обратно в таблицу

**На каждую регистрацию:**
- Записывает все данные в Registrations (ФИО, телефон, TG, возраст, опекун если несовершеннолетний, согласия)
- Генерирует правильный платёжный референс: `Имя – Event_ID`
- Ставит Retention_Until = дата события + 90 дней
- Защищает от дублей при повторной доставке вебхука (Submission_ID)

**Еженедельно:**
- Анонимизирует Phone_Number, Payment_Reference, Guardian_Name, Guardian_Phone по истечении срока хранения

## 5. Технические решения и почему они приняты именно так

### Триггер: installable onEdit, а не opros/cron
Simple onEdit (встроенный) не может вызывать UrlFetchApp — нет авторизации. Installable trigger — обязательно, не workaround. Подтверждено официально.

### Формат даты: parseDate() + formatDateForPost()
Sheets возвращает date-ячейки как JavaScript Date объект, не строку. Голое `String(dateObject)` даёт нечитаемый формат типа `Mon Jul 12 2026 10:00:00 GMT+0200 (Central European Summer Time)`. `formatDateForPost()` всегда даёт `12.07.2026 10:00` независимо от того, что лежит в ячейке — Date или текстовая строка.

### Защита от двойной публикации
Auto-trigger: если `Calendar_Link` уже заполнен → молча восстанавливает Status=Active, не постит повторно.
Ручной Retry (меню): всегда публикует — это осознанный выбор администратора, не случайная правка.

### JSONP вместо fetch() для сайта
Apps Script Web App отвечает внутренним HTTP-редиректом на `script.googleusercontent.com`. Заголовок `Access-Control-Allow-Origin: *` не всегда переживает этот редирект. `<script src="...?callback=fn">` в принципе не подчиняется CORS-политике — это не запасной вариант, а правильный инструмент для публичных данных с Google-сервисов.

### Один Web App URL для doGet и doPost
Apps Script автоматически маршрутизирует по HTTP-методу. doGet = данные для сайта, doPost = вебхук Tally. Один деплой, один URL для чек-листа.

### Секрет в URL вебхука, а не HMAC-подпись
Google официально подтвердил (issue tracker, 2023): Apps Script Web App не даёт доступ к заголовкам входящего запроса. Проверить `Tally-Signature` из HTTP-заголовка физически невозможно. Секрет в URL параметре — единственный вариант в рамках платформы.

### Кэш 5 минут + инвалидация при публикации
При публикации нового события кэш сбрасывается немедленно — сайт показывает его сразу. 5-минутный кэш защищает от избыточных чтений Sheets при высоком трафике сайта.

### colorId как число, а не строка
Официальный пример Google Calendar Advanced Service требует число. Строка `'11'` вместо `11` может вызвать API-ошибку на некоторых версиях.

## 6. Анализ ограничений и рисков

### Технические

**Время выполнения скрипта (~4–7 сек на событие)**
Квота Apps Script: 6 минут на одно выполнение. Даже при медленных ответах API запас огромный. Риска нет.

**Одноязычные описания событий на сайте**
Title и Description в Sheets — одно поле. Сайт показывает текст в том языке, в котором написано. Решения: (а) писать в одном языке (фактически так и работает), (б) договориться на EN как языке анонсов, (в) ввести формат `RU:... | EN:...` и парсить в doGet. Сейчас — вариант (а), как задумано.

**Revolut.me лимиты**
Personal ссылка: ~20 платежей/неделю (зависит от тарифного плана). При пиковых неделях (старт сезона, когда многие платят разом) возможно превышение. Мониторить в Revolut-приложении. При систематическом превышении — переход на Revolut Business Payment Links.

**Facebook App Review**
Занимает 1–4 недели, требует Business Verification (документы IČO). До этого система полностью работоспособна через ручной copy-paste из Telegram DM. Параллельный трек, не блокирует ничего.

**Telegram API лимиты**
30 сообщений/сек в канал. При 30 событиях/месяц мы делаем 30 запросов суммарно (не 30/сек). Риска нет.

### Бизнес-логика

**Нет контроля вместимости**
Tally (Free) может закрыть форму после N ответов — это встроенная бесплатная функция (Settings → Close form → After N responses). Настраивать её нужно вручную при создании формы для событий с лимитом мест. Автоматизировать через скрипт можно, но это требует Tally API (только PRO).

**Нет подтверждения регистрации на email участнику**
Respondent email notifications в Tally = только PRO (~$29/мес). Бесплатная альтернатива: Self notification (тебе на почту при каждой регистрации) — это FREE. Включить в Tally: Form Settings → Email notifications → Self email notifications. Это даёт тебе мгновенный алерт на почту и возможность вручную написать участнику если нужно.

**Напоминание перед событием — реализовано**
`sendEventReminders()` — ежедневный time-driven триггер, сравнивает календарную дату события (Europe/Prague) с "завтра" и шлёт напоминание в Telegram-канал за день до события. Corporate-события исключены (не публикуются). Известный остаточный риск: если Apps Script по какой-то инфраструктурной причине пропустит выполнение триггера в конкретный день (редко, но документированно случается), напоминание для события на следующий день не уйдёт — компенсирующего повторного прогона нет. Для этого масштаба (~30 событий/мес) это приемлемый риск, не критичный сбой.

**Сверка платежей — полностью вручную**
Участник платит в Revolut, ты видишь в приложении. Сверяешь с Registrations. Это не автоматизируется без Revolut Business API (платный тариф + OAuth). Единственная реально ручная операция с регулярной нагрузкой — ~5–10 мин/день.

**Нет уведомления при отмене события**
Если событие отменяется — нужно вручную написать в Telegram и вручную удалить событие из Calendar. Автоматизировать: при Status = Archived → скрипт удаляет Calendar-событие и постит уведомление в канал. Не реализовано сейчас.

## 7. Функции Tally, которые стоит включить прямо сейчас (бесплатно)

**Self email notification (уведомление тебе при каждой регистрации)**
Form Settings → Email notifications → Self email notifications → ON
Это бесплатно. Каждая новая регистрация сразу падает тебе на почту — имя, телефон, TG, что за событие.

**Close form after N responses (лимит мест)**
Form Settings → Close responses → After N responses
Бесплатно. Для событий с лимитом (например, HEMA-тренировка на 12 человек) — закрывай форму после нужного числа заявок.

**reCAPTCHA (защита от спама)**
Form Settings → reCAPTCHA → ON
Бесплатно. Особенно важно для бесплатных событий, где нет финансового барьера.

## 8. Что нужно сделать один раз при запуске (итоговый порядок)

1. Создать Telegram-бота через @BotFather → добавить в канал → получить токены
2. Google Calendar → создать публичный клубный календарь → скопировать Calendar ID
3. Google Sheets → импортировать `events-registrations-template.xlsx`
4. Extensions → Apps Script → вставить `Code.gs` → заменить `appsscript.json` → заполнить CONFIG
5. Services → добавить Google Calendar API (v3)
6. Запустить `onOpen` → пройти авторизацию
7. Triggers → `onStatusChange` (On edit) + `cleanupOldRegistrations` (Weekly)
8. Tally → создать форму согласно `tally-form-structure.md` (Hidden fields, conditional logic)
9. Tally → Form Settings → Self email notifications → включить
10. Deploy → New deployment → Web App → скопировать URL
11. Tally → Integrations → Webhooks → вставить URL + `?secret=ТВОЙ_СЕКРЕТ`
12. В index.html → подставить `EVENTS_API_URL` → закоммитить в GitHub
13. Google Calendar → Settings → Access permissions → Make available to public
14. Тест: строка в Events → Trigger_Sync → проверить Calendar / Telegram / Sheets
15. Тест: пройти Tally форму → проверить Registrations, Photo_Consent = no
16. Параллельно: Meta App Review + Business Verification (фоновый трек)

## 10. Итоговая оценка системы

**Что реально хорошо:**
- Единственный ввод данных (одна строка в Sheets) запускает всё одновременно
- Нулевая стоимость навсегда — не "почти бесплатно", а буквально $0
- Каждый компонент заменяем независимо (сломался Tally → подключи Google Forms через `doPost`)
- Error handling: ошибка одного канала не блокирует другие, `Sync_Error` виден прямо в таблице
- GDPR покрыт реально, не для галочки: правовое основание верное, срок хранения автоматический, Guardian-поля анонимизируются

**Что принято как осознанный компромисс, не как недостаток:**
- Ручная сверка платежей Revolut — цена отказа от Stripe/платного эквайринга
- Одноязычные описания событий — цена реального zero-maintenance (4 поля × 30 событий = 120 переводов/мес без автоматизации)
- Facebook на ручном copy-paste до Meta App Review — цена отказа от Make.com

**Что стоит добавить при росте нагрузки:**
- Reminder-триггер (напоминание за 2 дня) — ~30 строк в Code.gs
- Archived-обработчик (удалить Calendar event при отмене) — ~20 строк
- Capacity field в Events (MAX_PARTICIPANTS) + автозакрытие Tally через API (требует Tally PRO)
