# Community Automation Hub — гайд по внедрению (только Apps Script, $0 навсегда)

Никакого Make.com. Один Apps Script-проект, привязанный к таблице, делает всё: создаёт события в календаре, шлёт в Telegram, постит в Facebook (когда будет одобрено), принимает регистрации Tally напрямую через свой же URL, и чистит персональные данные по таймеру.

## Почему это гарантированно $0
- **UrlFetchApp** (исходящие HTTP-запросы к Telegram/Facebook API): квота 20 000 вызовов/день на обычном Google-аккаунте — мы используем ~3–4 на событие и 0 на регистрацию
- **Суммарное время выполнения триггеров**: 90 минут/день — у нас задачи занимают секунды
- **Google Calendar API**: квота по умолчанию 1 000 000 запросов/день
- **Apps Script Web App** (приём вебхука от Tally) — без ограничения по числу вызовов, без какой-либо платной составляющей

Нигде в этой цепочке нет понятия "кредиты" или "лимит тарифа" — в отличие от Make, тут физически нечему заканчиваться.

## Шаг 1. Структура Google Sheets

### Лист `Events`
`Event_ID | Status | Community | Category | Title | Start_DateTime | End_DateTime | Location_Name | Location_Address | Price_CZK | Description | Tally_Form_URL | Telegram_Post_ID | Calendar_Link | Last_Synced_At | Sync_Error`

Колонки `Tally_Form_URL`, `Telegram_Post_ID`, `Calendar_Link`, `Last_Synced_At`, `Sync_Error` заполняются скриптом автоматически — оставь их пустыми при создании.

`Status` — выпадающий список: `Draft`, `Active`, `Archived`, `Trigger_Sync`.

### Лист `Registrations`
`Reg_ID | Event_ID | Timestamp | Full_Name | Telegram_Handle | Phone_Number | Payment_Reference | Retention_Until`

## Шаг 2. Telegram-бот

1. В Telegram открой `@BotFather` → `/newbot` → задай имя и username (должен заканчиваться на `bot`) → получишь **токен**
2. Добавь бота администратором в канал: Настройки канала → Administrators → Add Admin → найди бота по username → дай право "Post Messages"
3. **Chat ID канала**: если у канала публичный username — просто используй `@имя_канала` как chat_id, без числового ID
4. **Chat ID админа** (личка для буфера WhatsApp/Facebook): напиши боту `/start` в личке, затем открой в браузере
   `https://api.telegram.org/bot<ТВОЙ_ТОКЕН>/getUpdates`
   и найди своё числовое `chat.id` в ответе

## Шаг 3. Google Calendar

1. Создай отдельный календарь (не личный): Google Calendar → "Other calendars" → "Create new calendar" — так его можно публично шарить, не раскрывая личные события
2. Settings этого календаря → Access permissions → включи "Make available to public" → "See all event details"
3. В том же разделе Settings → "Integrate calendar":
   - скопируй **Calendar ID** (вид `xxxxx@group.calendar.google.com`)
   - скопируй **Public URL to this calendar** — это будет ссылка "все события" в постах

## Шаг 4. Tally-форма (одна на все события, с условной логикой)

Полная структура полей, скрытых параметров и текстов согласий (включая ветвление по возрасту и по категории события) — в отдельном файле `tally-form-structure.md`. Здесь — короткая версия:

1. Создай форму с полями: `Full_Name`, `Telegram_Handle`, `Phone_Number`, `Age`
2. Hidden fields (заполняются автоматически из URL, не трогать вручную): `event_id`, `title`, `category`, `community`, `price`
3. Условная логика в Tally (Settings → Logic): если `Age < 18` → показать блок опекуна; если `category` = "HEMA Training" → показать вейвер про спарринг, иначе → общий вейвер
4. Используй готовые тексты согласий (фото/видео, GDPR-уведомление, вейверы) из `tally-form-structure.md` — они уже согласованы с твоими существующими документами, не придумывай новые формулировки
5. Запомни **Form ID** — часть ссылки формы после `/r/`

## Шаг 5. Apps Script проект

1. Открой таблицу (Events+Registrations) → **Extensions → Apps Script**
2. **Project Settings** (шестерёнка слева) → Time zone → `Europe/Prague`
3. **Services** (значок "+" слева) → Add a service → найди и добавь **Google Calendar API**

Открой файл `Code.gs`, который шёл отдельным файлом вместе с этим гайдом — это единственная актуальная версия кода (раньше код дублировался прямо в этом гайде текстом, но так проще двум копиям разойтись при правках, поэтому теперь код живёт только в одном файле). Скопируй его содержимое целиком в редактор Apps Script и заполни блок `CONFIG` в начале своими значениями.

**Важные нюансы кода (проверено по официальной документации):**
- `Calendar.Events.insert` — это не CalendarApp, а **Advanced Service** (отсюда и шаг с подключением Google Calendar API) — только так можно передать `colorId` и получить нормальную `htmlLink` на событие.
- `e.parameter.secret` в `doPost` — это не временное упрощение, а единственный вариант: Google официально подтвердил (issue tracker, 2023), что Apps Script Web App **не даёт и не будет давать доступ к заголовкам входящего запроса** по соображениям безопасности. Поэтому штатный механизм Tally — подпись `Tally-Signature` (HMAC-SHA256 в заголовке) — здесь физически нельзя проверить, и секретный параметр в самом URL вебхука — это не «как смогли», а правильное решение в рамках платформы.
- `byLabel['Full_Name']` и т.д. — сопоставление идёт **по тексту названия (label) поля** в форме Tally, а не по случайному `key`. Назови вопросы в форме ровно так, как в `Code.gs`, либо поправь эти строки под свои реальные названия после первого тестового сабмишена.
- **Чекбоксы Tally возвращают массив ID выбранных опций, а не `true/false`** (если не отмечено — `[]`, если отмечено — `["uuid"]`). Поэтому в коде есть отдельная функция `isChecked()` — простая проверка `значение ? ... : ...` здесь дала бы неправильный результат, потому что пустой массив `[]` в JavaScript считается истинным значением.
- Если в форме появится поле типа Dropdown/Multiple Choice/Multi-select (например, размер футболки) — для него Tally тоже присылает не текст, а ID выбранной опции, а текст лежит отдельно в массиве `options` того же поля. Под текущие 8 полей формы (текст, число, чекбоксы) это не нужно, но если будешь добавлять поле с выбором из списка — учти этот момент при правке `doPost`.
- `Submission_ID` в `Registrations` — защита от дублей: Tally повторяет доставку вебхука с интервалами 5 мин / 30 мин / 1 ч / 6 ч / 1 день, если не получит ответ 2xx в течение 10 секунд. Для текущих объёмов скрипт отвечает за секунды, так что это редкий случай (сетевой сбой при доставке самого ответа), но раз нашёл — добавил проверку, а не оставил как теоретический риск.

## Шаг 6. Авторизация скрипта

1. В редакторе Apps Script выбери функцию `onOpen` в выпадающем списке наверху → **Run**
2. Появится экран Google "это приложение не проверено" — это нормально для твоего собственного скрипта, который ты не публикуешь. Жми **Advanced** → **Go to [имя проекта] (unsafe)** → **Allow**
3. Это разово выдаёт права на Sheets, Calendar и внешние запросы (UrlFetchApp)

## Шаг 7. Триггеры

В редакторе Apps Script — значок часов слева ("Triggers") → **Add Trigger**:

**Триггер 1 — рассылка событий**
- Function: `onStatusChange`
- Event source: `From spreadsheet`
- Event type: `On edit`

**Триггер 2 — очистка персональных данных**
- Function: `cleanupOldRegistrations`
- Event source: `Time-driven`
- Type: `Week timer`, день и время — например, понедельник, 3:00

⚠️ **Важно, раз у тебя теперь подключены Google Drive / Calendar коннекторы в Claude:** триггер `On edit` срабатывает только когда строку правит человек руками через интерфейс Google Sheets. Если строка в `Events` будет изменена программно — через Sheets API, через скрипт другого проекта, или если меня (Claude) попросить отредактировать ячейку через коннектор — `Status = Trigger_Sync` не запустит автоматизацию, это официально подтверждённое ограничение Apps Script. В таком случае нужно вручную выделить ячейку в этой строке и выбрать в меню таблицы **Community Hub → 🔄 Повторить синхронизацию для текущей строки**.

## Шаг 8. Деплой Web App (приём вебхука от Tally)

1. **Deploy → New deployment** → тип: **Web app**
2. Execute as: **Me**
3. Who has access: **Anyone**
4. **Deploy** → скопируй полученный URL (заканчивается на `/exec`)
5. В Tally: форма → **Integrations → Webhooks → Add webhook** → Endpoint URL:
   `ТВОЙ_URL_СО_ШАГА_4?secret=ТВОЙ_TALLY_WEBHOOK_SECRET`
6. **При любых будущих правках кода**: не создавай новый деплой — используй **Deploy → Manage deployments → ✏️ → Version: New version → Deploy**. Так URL вебхука остаётся тем же и не нужно перенастраивать Tally каждый раз.

## Шаг 9. Facebook — отдельный параллельный трек

Не блокирует запуск всего остального:
1. Meta for Developers → создать приложение, запросить `pages_manage_posts` + `pages_read_engagement` + `pages_show_list` → App Review (описание use case + видео-демо, обычно 1–4 недели)
2. Business Verification через Meta Business Manager (документы IČO)
3. Получить **никогда не истекающий** Page Access Token: short-lived user token → long-lived user token (~60 дней) → Page token, полученный из long-lived токена, не истекает
4. В `CONFIG` поставить `FACEBOOK_ENABLED = true`, заполнить `FACEBOOK_PAGE_ID` и `FACEBOOK_PAGE_TOKEN`

До этого момента текст для Facebook просто падает админу в Telegram-буфер вместе с WhatsApp-текстом (Шаг 5, блок "Telegram: буфер админу") — система полностью рабочая без Facebook с первого дня.

## Шаг 10. Тестирование (по порядку)

1. Добавь тестовую строку в `Events`, заполни все поля, поставь `Status = Trigger_Sync`
2. Проверь: появилось событие в календаре с правильным цветом → пришёл пост в Telegram-канал с кнопкой → пришёл буфер админу в личку → в строке появились `Calendar_Link`, `Telegram_Post_ID`, `Last_Synced_At`, `Status` стал `Active`
3. Если что-то не сработало — смотри `Sync_Error` в строке и вкладку **Executions** в редакторе Apps Script (там полный traceback)
4. Исправь причину, выдели любую ячейку в этой строке и используй меню **Community Hub → 🔄 Повторить синхронизацию для текущей строки**
5. Перейди по `Tally_Form_URL` из тестовой строки, заполни и отправь форму → проверь, что в `Registrations` появилась строка с правильным `Event_ID` и `Retention_Until`

## Чек-лист GDPR (без раздела про конкурс)
- В форме Tally — уведомление об обработке данных + добровольные чекбоксы (фото/видео, рассылка анонсов) — тексты и условная логика в `tally-form-structure.md`, основаны на твоём реальном документе `GDPR_Zpracování_osobních_údajů_2026` (основание — исполнение договора, не согласие, поэтому обязательная галочка на сам факт обработки не нужна)
- Срок хранения и автообезличивание — уже реализовано через `Retention_Until` + еженедельный триггер `cleanupOldRegistrations`
- Доступ к самой таблице — ограничить (Share только себе/доверенному админу, не "всем по ссылке")
- В буфер админу уходит только публичная информация о событии — не персональные данные регистрантов
